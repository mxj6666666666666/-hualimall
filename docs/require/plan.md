# 华立商城后端详细实施计划（基于当前仓库）

## 1. 问题与现状
- 当前仓库已具备商品与订单主链路：`Controller -> Service -> Mapper -> XML`，并使用 `Result<T>` / `PageResult<T>` 统一返回。
- 已落地接口：商品查询、管理端商品 CRUD、订单创建/列表/详情/取消。
- 现阶段主要缺口：用户认证、购物车、分类、订单状态流转完善、统一异常处理、接口文档与可交付资料（SQL/README）。
- `docs\require` 中目标范围明显大于当前代码，需要分阶段补齐，避免一次性扩展导致质量下降。

## 2. 目标与范围（本计划默认）
- 目标：在现有代码基础上，交付“可演示、可运行、可说明”的最小完整商城后端。
- 范围内：
  - 用户模块：注册/登录/个人信息（JWT）。
  - 商品模块：保留现有能力，补齐分类与查询过滤能力。
  - 购物车模块：增删改查 + 下单前数据来源。
  - 订单模块：对接购物车、状态流转、库存扣减事务。
  - 管理端：商品管理增强 + 订单状态更新。
  - 基础能力：全局异常、鉴权拦截、统一错误码、接口文档。
- 范围外（当前阶段不做）：
  - 真实支付、优惠券、秒杀、复杂 RBAC、消息队列。

## 3. 实施策略
- 采用“先稳定主链路，再横向补模块”的方式：
  1. 先修复并稳定现有订单/商品链路的边界行为（事务、状态、参数校验）。
  2. 再补用户与购物车，确保订单创建不再依赖前端传全量订单结构。
  3. 最后增强管理端与交付物（文档、初始化脚本、演示配置）。
- 所有新增接口继续沿用当前仓库约定：
  - `controll` 包名保持不变；
  - `Result.success(...)` 统一返回；
  - Mapper 接口与 XML 参数名严格对齐（尤其 `@Param` 与 `foreach collection`）。

## 4. 分阶段任务

### Phase A：基线治理（先把现有能力做“稳”）
1. 校正与补强参数校验：
   - 对分页参数、ID、数量等做最小合法性校验。
2. 订单核心逻辑加固：
   - 下单流程引入事务边界；
   - 明确订单状态枚举及状态转换约束；
   - 对商品不存在、库存不足、订单不存在等错误返回统一业务错误。
3. 持久层一致性检查：
   - 对照 `OrderMapper.java` / `OrderMapper.xml`、`ProductMapper.java` / `ProductMapper.xml` 参数名一致性；
   - 保持批量接口 `collection` 命名与注解一致。

### Phase B：补齐用户与鉴权
1. 新增用户数据模型与接口：
   - 注册、登录、`/user/info`。
2. JWT 能力：
   - token 签发与解析；
   - 登录态拦截器（对需鉴权接口生效）。
3. 兼容现有接口：
   - 订单创建优先从登录态读取用户身份，不再依赖前端任意传 `userId`。

### Phase C：补齐购物车与下单入口
1. 新增购物车表与实体/Mapper/Service/Controller：
   - `/cart/list`、`/cart/add`、`/cart/update`、`/cart/remove/{id}`。
2. 下单改造：
   - 从购物车“已选商品”生成订单；
   - 下单成功后清理已下单购物车项。
3. 一致性与并发：
   - 库存扣减使用受影响行数校验，防止超卖（简化方案）。

### Phase D：商品查询增强与管理端完善
1. 商品查询增强：
   - 在 `ProQueryParams` 增加可选筛选字段（分类、状态、关键词）；
   - 在 `ProductMapper.xml` 启用动态 where 条件。
2. 管理端订单管理：
   - 新增 `/admin/order/{id}/status`；
   - 约束可变更状态范围。
3. 兼容性：
   - 保留当前已有接口路径，新增能力尽量向后兼容。

### Phase E：交付与工程化收口
1. SQL 交付：
   - `schema.sql`（建表）+ `data.sql`（基础演示数据）。
2. 文档交付：
   - README 启动说明、测试账号、接口说明、已实现功能清单。
3. 接口文档：
   - 引入 Swagger/Knife4j（如与当前依赖版本兼容）或提供 Markdown API 文档。

## 5. 关键风险与应对
- 风险1：当前项目同时引入 JPA 与 MyBatis，职责混淆。
  - 应对：本阶段业务坚持 MyBatis 主路径，JPA 不参与新增业务逻辑。
- 风险2：订单下单与库存扣减一致性不足。
  - 应对：事务 + 扣减结果校验 + 明确异常返回，先满足单体可用的一致性。
- 风险3：接口命名和文档存在历史差异。
  - 应对：以现有代码路径为准，文档中提供“当前实现 vs 目标接口”的映射表。

## 6. 里程碑完成标准
- M1：现有商品/订单接口稳定，异常返回统一，分页与参数行为可预期。
- M2：用户登录与 JWT 全链路可用。
- M3：购物车到下单链路打通（购物车 -> 订单 -> 明细 -> 库存）。
- M4：管理端订单状态管理可用，商品筛选可用。
- M5：README + SQL + 接口文档齐备，可一键启动演示。

## 7. 执行说明
- 先按 Phase A -> E 顺序实现；每阶段完成后执行现有 Maven 测试/构建命令进行回归。
- 若你确认范围，我将按该计划把任务拆解为可执行 TODO 并进入实施。
