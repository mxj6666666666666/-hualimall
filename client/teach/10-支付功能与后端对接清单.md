# 支付功能后端对接清单（你本人在 `server` 完成）

> 说明：前端已在 `client` 增加支付页面与调用入口。  
> 本文只告诉你后端该怎么做，**我不会修改 `server` 代码**。

## 1. 你需要提供的后端接口

前端当前调用以下接口（统一响应结构：`{ code, message, data }`）：

1. `POST /payments`：创建支付单  
2. `GET /payments/{id}`：查询支付单详情（按支付单 ID）  
3. `GET /payments/order/{orderId}`：按订单查询当前支付单  
4. `PUT /payments/{id}/close`：关闭支付单

### 1.1 创建支付单

**请求体**
```json
{
  "orderId": 10001,
  "channel": "ALIPAY"
}
```

`channel` 建议枚举：`ALIPAY | WECHAT`

**成功返回 data 示例**
```json
{
  "id": 9001,
  "paymentNo": "P2026050816300001",
  "orderId": 10001,
  "channel": "ALIPAY",
  "status": "PENDING",
  "payUrl": "https://openapi-sandbox.example/pay/xxx",
  "codeUrl": "https://openapi-sandbox.example/qrcode/xxx",
  "expireTime": "2026-05-08T16:45:00+08:00"
}
```

> 前端会优先使用 `payUrl` 或 `codeUrl` 打开收银台/二维码。

### 1.2 查询支付单详情

`GET /payments/{id}` 返回同上结构，重点字段：
- `status`: `PENDING | PROCESSING | SUCCESS | CLOSED | FAILED`
- `expireTime`: 支付过期时间

### 1.3 按订单查询支付单

`GET /payments/order/{orderId}`

建议行为：
- 有支付单：返回最新有效支付单
- 没有支付单：返回业务错误（如 404 或自定义 code），前端会允许用户重新创建支付单

### 1.4 关闭支付单

`PUT /payments/{id}/close`

建议规则：
- 仅 `PENDING/PROCESSING` 可关闭
- 已成功支付不可关闭（返回明确错误）
- 关闭后更新支付状态为 `CLOSED`，并同步订单状态（若你的订单规则要求）

## 2. 订单与支付状态机（必须落地）

## 2.1 订单状态（当前前端约定）
- `0`: 待支付
- `1`: 已支付
- `2`: 已取消
- `3`: 已完成

## 2.2 支付状态（建议）
- `PENDING`: 待支付
- `PROCESSING`: 支付处理中
- `SUCCESS`: 支付成功
- `CLOSED`: 已关闭
- `FAILED`: 支付失败

## 2.3 状态流转建议
1. 创建支付单：`PENDING`
2. 用户在收银台支付中：`PROCESSING`（可选）
3. 回调确认成功：`SUCCESS`，订单改为 `1(已支付)`
4. 超时未支付或主动关闭：`CLOSED`
5. 支付渠道明确失败：`FAILED`

---

## 3. 第三方支付（沙箱）必须做的点

## 3.1 支付宝/微信下单
- 使用沙箱商户配置（AppId、商户号、私钥、公钥）
- 生成平台下单参数并获取 `payUrl` / `codeUrl`
- 将支付平台交易号保存到支付单（如 `channelTradeNo`）

## 3.2 异步回调通知
- 提供回调接口（例如 `/payments/notify/alipay`、`/payments/notify/wechat`）
- 严格验签：签名失败直接拒绝
- 验证金额、商户号、订单号、交易状态
- 回调处理要幂等（重复通知只能成功一次）

## 3.3 主动查询兜底
- 定时任务扫描 `PENDING/PROCESSING` 支付单
- 超过阈值主动调用三方查询接口
- 与本地状态对齐，避免长时间“处理中”

---

## 4. 幂等与一致性（重点）

## 4.1 创建支付单幂等
- 同一订单在“未终态支付单存在”时，建议复用原支付单而不是重复创建
- 可用 `(order_id, status in PENDING/PROCESSING)` 唯一约束或业务锁

## 4.2 回调幂等
- 以 `channelTradeNo` + `tradeStatus` 做去重
- 已处理成功的支付单再次收到成功回调时直接返回成功，不重复改订单

## 4.3 金额一致性
- 回调金额必须与订单应付金额一致
- 金额不一致时标记风控异常并拒绝入账

## 6. 与前端联调步骤（按这个顺序）

1. 先打通 `POST /payments`，让前端可拿到 `payUrl/codeUrl`
2. 打通 `GET /payments/{id}`，前端轮询可见状态变化
3. 打通回调，确保支付成功后订单状态从 `0 -> 1`
4. 打通 `PUT /payments/{id}/close`，验证未支付关闭逻辑
5. 打通 `GET /payments/order/{orderId}`，支持页面刷新恢复支付单

---

## 7. 验收用例（建议你自己跑）

1. 正常支付成功：订单 `0 -> 1`，支付单 `PENDING -> SUCCESS`
2. 重复支付通知：系统只更新一次，不重复扣减/重复写状态
3. 超时未支付：支付单自动 `CLOSED`，订单仍为待支付或按业务关闭
4. 人工关闭后再支付：三方回调到达时应拒绝状态回滚
5. 金额篡改回调：验签或金额校验失败，记录日志并拒绝
